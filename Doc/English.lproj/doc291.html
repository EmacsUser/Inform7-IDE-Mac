20.2. Sam Ashwell - [i7] upper limit on unnamed values

Reported by: Sam Ashwell / Date: 11 August 2005

Nature: The source looks legal, and generates no Problem messages, but then generates errors when fed into I6

Build: 2V60

Platform: Mac OS X

Code example:

"test" by Sam Kabo Ashwell

First Room is a room.

A thing is either outsider or clan member.
A thing is either male or female.
A thing is either married or single.
A thing can be enfranchised or disenfranchised.
A thing can be riparian or arid.
A thing can be irrigated or dry.
things can be poisonous or poisonless.
things can be aquatic or terrestial.
things can be scavenging or aloof.
things can be predatory or snuggly.
things can be malign or innocent.
things can be dangerous or safe.
things can be large or tiny.
things can be wild or domestic.
Things can be edible or inedible.
A thing is either thirsty or thirstless.
A thing is either local or exotic.

What goes wrong:

I seem to have hit a limit.

If I use over 16 unnamed values (as in the code above) the game

compiles, but crashes the interpreter on loading; if I use 16 or less,

the game will load happily. This behaviour doesn't vary between z5 and

z8 files. Similarly, the behaviour doesn't change if the unnamed

values are assigned between different kinds, or to different

individual objects.

Comments:

Clearly this is Bad Coding and I should be creating new named values;

it's not a big problem to work around.

It'd be nice, therefore, to prevent a game with 18 unnamed values from

compiling, with an error message to the tune of

'I'm afraid you've reached the limit for unnamed values. Inform only

allows 16 conditions of the form 'A fellow (can be / is either)

delightful or ennervating'. To work around the problem I suggest

naming your values, thus: 'Waggery is a kind of value. A fellow has

waggery. The waggeries are delightful and ennervating.'

There is no limit (probably) on this sort of value, and using them is

better practise.'
